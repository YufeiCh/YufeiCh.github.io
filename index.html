toydb 架构

从整体上的高层次来看，toydb 的架构为多个根据复制状态机来执行 SQL 事务的节点组成的集群。客户端可以连接到集群中的任何一个节点并执行 SQL 查询。toydb 旨在提供线性一致性即强一致性与串行化，但是目前由于只实现了快照隔离级别所以略显不足。

RAFT 算法被用于 toydb 的集群共识，该算法能够保证在集群中的大多数节点（半数节点）都可用时，可以忍受任意数量的节点失效。当一个节点背选举为 leader 时，需要复制 RAFT command 到其他节点上，并且其他节点需要将这些RAFT command到状态机中。当 leader 掉线之后，一个新的 leader 会被选举出来，使得整个集群可以继续运行。Client command 会被直接发送给 leader。

整个架构介绍将从更高层次的整体总览上开始，然后再自底向上的介绍每一个组件。在介绍的同时，将注意到设计选择与权衡。

节点组件

一个 toydb 节点由以下三个组件组成：

  • 存储引擎：在磁盘与内存中存储数据，管理事务
  • RAFT 共识引擎：处理集群协作与状态机的复制
  • SQL 引擎：解析 SQL 语句，生成执行计划，并执行。

这些组件被集成于 toydb server 中，此外 toydb server 需要处理与 client 以及其他节点之间的网络连接。下图描述了 toydb 的内部结构：

最底部为一个简单的键值存储，其中存储了所有的 SQL 数据，并且被包装在一个 MVCC 的键值存储中来添加 ACID 的事务。在 MVCC 存储的上方是一个 SQL 存储引擎，提供简单的用于表、行以及索引的 CRUD 操作。这些组成了节点的核心存储引擎。

SQL 存储引擎被包装于一个 RAFT 状态机的接口中，使得上层的 RAFT 共识引擎可以管理底层的存储引擎。每一个 RAFT 节点从 clients 那里接收 commands，并与集群中的其他 RAFT 节点进行协作来在一个顺序的 command 日志上达成共识。一旦 command 被提交到日志中，就会被发送给状态机驱动器来应用到本地状态机中。

在 RAFT 引擎的上方是一个基于 RAFT 并实现了 SQL 存储接口的 SQL 存储引擎，这个引擎会提交 commands 到 RAFT 集群中。这个机制允许剩余的 SQL 层得以像使用本地存储一样直接使用 RAFT 集群。SQL 引擎 管理来自 clients 的 SQL 会话，在这里将 SQL 查询直接当成文本来解析，生成执行计划，并根据 SQL 存储引擎来执行。

除了上面这些组件以外，toydb server 还需要处理网络通信、配置、日志打印以及其他进程级别的问题。

存储引擎

存储引擎实际上有两种不同的类型：一种是被上层的 SQL 引擎使用的键值存储类型，一种是被 RAFT 节点使用的 log-structured 存储。这些都可以通过 storage_sql 和 storage_raft 配置选项来可插拔使用，并且跟还有不同的特性有多种实现方式。

SQL 存储引擎在 SQL 章节再进行介绍。

  • 为了性能表现，所有的数据应当按照键的顺序来存储
  • 键应该使用保持顺序的字节编码，以允许范围扫描

底层存储实际上并不关心它存储了什么数据，但是这个模块为了供上层使用而提供了一种按需的键编码。这些存储层通常使用由几个可变长度值组成的复合键 (例如，一个索引键包括表，列与值)，并且每个键的自然顺序也需要被保留，满足这些要求的编码如下：

  • bool: 0x00 为 false，0x01 为 true
  • Vec<u8>: 以 0x0000 结尾，0x00 作为 0x00ff 逃逸
  • String: 类似 Vec<u8>
  • u64: 大端编码
  • i64: 大端编码，符号位翻转
  • f64: 大端编码，符号位翻转如果为整数，全部翻转如果为负数
  • sql::Value: 类似前面的描述，加上类型前缀，0x00=Null, 0x01=Boolean, 0x02=Float, 0x03=Integer, 0x04=String

默认的键值存储为 storage::kv::Memory，是一个内存型的B+树实现，这是一种每个节点有多个键的搜索树变体(为了利用缓存位置) ，并且只在叶节点中有值。随着键/值对的添加和删除，树节点被拆分、合并和旋转，以保持它们的平衡和至少半满。

虽然键/值数据存储在内存中，但是toyDB 通过 Raft log提供了持久性，这些log会被保存到磁盘上。在 toydb 启动时，将重播 Raft 日志以恢复内存中的存储。

键值权衡

内存存储：将键/值数据存储在内存中比磁盘存储具有更好的性能并且更易于实现，但要求数据集适合内存。对于大型数据集，在启动时重放 Raft 日志也可能需要相当长的时间。然而，由于 toyDB 数据集预计很小，这在很大程度上是有利的。

字节序列化：由于主存储在内存中，（反）序列化键/值对会增加大量不必要的开销。不过一开始并不清楚toyDB会使用内存存储，byte slices是一个简单的接口，不管存储介质如何都可以使用。

B+树扫描：B+树通常在相邻的叶节点之间有指针，以便更有效地进行范围扫描，但 toyDB 的实现没有。这会使实现复杂化，并且在随机访问延迟较低的内存中，性能优势通常不会那么大。然而，这与其他实现细节一起导致范围扫描是 O(log n) 而不是每步 O(1)。

密钥编码：不使用任何压缩，例如可变长度整数，实现上倾向于选择简单性和正确性。

MVCC 事务

MVCC（Multi-Version Concurrency Control）是一种相对简单的并发控制机制，它为 ACID 事务提供快照隔离，无需取锁或写阻塞读。 它还对所有数据进行版本化，允许查询历史数据。

toyDB 在键/值层将 MVCC 实现为 storage::kv::MVCC，任何实现了 storage::kv::Store 的存储引擎都可以作为 MVCC 的底层存储。 begin 返回一个新的事务，它提供了常用的键/值操作，例如 get、set 和 scan。 此外，它还有一个 commit 方法可以持久化更改并使更改对其他事务可见，还有一个 rollback 方法可以丢弃更改。

当一个事务开始时，从 Key::TxnNext 获取下一个可用的事务 ID 并递增该 ID，然后通过 Key::TxnActive(id) 将当前事务记录为活跃事务。这个事务还会获取一个快照，其中包含事务开始时所有其他活跃事务的 ID，并将其保存为 Key::TxnSnapshot(id)。

键/值对保存为 Key::Record(key, version)，其中 key 是用户提供的键，version 是创建记录的事务 ID。 事务的键/值对的可见性如下：

  • 对于给定的用户键，从当前事务的 ID 开始反向扫描 Key::Record(key, version)。
  • 跳过快照中活跃事务 ID 列表中所有版本。
  • 返回第一个匹配的记录，返回 Some(value)。 如果键被删除，那就返回 None。

写入键/值对时，事务首先通过扫描对它不可见的 Key::Record(key, version) 来检查是否有任何冲突。 如果找到，则返回序列化错误，客户端必须重试事务。 否则，事务将写入新记录并以 Key::Update(id, key) 的形式跟踪更改，以防以后必须回滚。

当事务提交时，它只是删除其 Txn::Active(id) 记录，从而使其更改对任何后续事务可见。如果事务需要回滚，它会遍历所有 Key::Update(id, key) 条目，并在删除其 Txn::Active(id) 条目之前，删除已经写入的键/值记录。

这个简单的方案足以通过快照隔离提供 ACID 事务保证：提交是原子的，事务看到键/值存储在事务开始时的一致快照，任何写入冲突都会导致串行化错误，必须重试。

为了满足时间旅行查询，只读事务只需加载过去事务的快照，并像使用与普通事务一致的可见性规则。

MVCC 权衡

只读事务 ID：所有事务，甚至是只读事务，都被分配一个唯一的事务 ID。这意味着单个独立的 SELECT 查询都会产生写入操作以增加事务 ID 计数器，这个对性能的影响可能是很大的。

串行化：快照隔离不是完全串行化的，因为表现出写倾斜异常。这将需要串行化的快照隔离，这在第一个版本中被认为是不必要的——可能会在以后实现。

垃圾回收：永远不会删除旧的 MVCC 版本，从而导致只增不减的磁盘使用。但是这也保留了完整的数据历史，并简化了实现。

事务 ID 溢出：事务 ID 会在 64 位后溢出，但 toyDB 永远不会发生这种情况。

Log-structured 存储

Raft 节点需要保留编码为任意字节片的状态机 command 日志。此日志是 append-only 的，将其存储在随机访问键/值存储中，将比使用专门为此访问模式构建的 Log-strutured 存储更慢且更复杂。

Raft 节点将所有接收到的 command 添加到其本地日志中，但只有在大多数节点达成共识后才提交command。本地日志可能需要被截断，例如在 leader 发生变化的情况下，删除一些未提交的条目。

此外，存储必须能够通过 set_metadata(key, value) 和 get_metadata(key) 方法为 Raft 节点存储少量任意键/值元数据对。

toyDB 中的默认日志存储是 storage::log::Hybrid，它将未提交的条目存储在内存中，将已提交的条目存储在磁盘上。这允许日志以仅追加方式写入并按顺序写入，从而为写入和批量读取提供非常好的性能。未提交条目的数量通常也很少，因为达成共识通常很快。

新日志条目保存在 VecDeque（双端队列）中，直到它们被提交。 提交时，条目会追加到带有 u32 长度前缀的文件中，并且文件会 fsynced（如果启用）。条目位置保存在内存中的 HashMap 中，该 HashMap 以条目索引为键，用于检索，并且该映射在启动时通过扫描日志文件重建。

元数据键/值对保存在内存中的 HashMap 中，并且整个 HashMp 在每次写入时都写入一个单独的文件。

Log的权衡

启动时的日志扫描：在启动时扫描整个文件以建立条目索引可能很耗时，而且索引也需要内存。然而，这避免了必须维护单独的索引存储，这对于 fsync 来说可能是昂贵的，并且 toydb 的数据集应该很小。

元数据存储：元数据键/值对应该存储在例如磁盘上的 B 树键/值存储，但 toyDB 当前没有这样的存储。然而，目前元数据项的数量非常少——特别是当前的 Raft term/vote 元组。

内存缓冲：如果共识被迫停滞，在内存中缓冲未提交的条目可能需要大量内存，例如 由于失去大多数节点。 然而，对于 toyDB 用例来说，这不是一个主要问题，并且它避免了必须执行额外的（也可能是随机的）磁盘 IO，从而大大提高了性能。

垃圾回收：没有旧日志条目的垃圾回收，所以日志会无限增长。然而，这是必要的，因为默认的 toyDB 配置默认使用内存中的键/值存储，并且没有其他持久化存储。

RAFT 共识引擎

Raft 共识协议在 Raft 原论文中有很好的解释，这里不再赘述。toyDB 的实现非常接近这篇论文。

Raft 节点 raft::Node 是实现的核心，它是一个有限状态机，具有节点角色的枚举：leader、follower 和 candidate。 这个枚举包装了 RoleNode 结构，它包含公共节点功能，并且对实现 Raft 协议的特定角色 Leader、Follower 和 Candidate 是通用的。

节点使用 ID 和peer ID 列表进行初始化，并通过传递 raft::Message 消息进行通信，通过 Node.step() 调用接收，通过 mpsc 通道发送。节点还使用逻辑时钟来跟踪例如选举超时和心跳，时钟通过 Node.tick() 调用定期计时。 这些方法是同步的，可能会导致状态转换，例如当 candidate 获得获胜选票时，将 candidate 转变为 leader。

节点有一个 command 日志 raft::Log，使用一个 storage::log::Store 进行存储。leader 通过 request messsage 接收客户端命令，将它们复制给其他节点，并根据共识将命令提交到日志。提交命令后，将其异步应用于状态机。

Raft 管理的状态机（即 SQL 存储引擎）实现了 raft::State 特性，并在初始化时提供给节点。状态机驱动程序 raft::Driver 拥有状态机的所有权，并在一个单独的线程（或者更确切地说，一个 Tokio 任务）中运行，通过 mpsc 通道接收指令——这避免了长时间运行的 comamnd 阻塞主 Raft 节点响应消息。

除了应用状态机命令外，驱动程序还通过 mpsc 的信息发送通道响应客户端请求。当 leader 收到来自客户端的状态变更请求时，它不仅将命令追加到其日志中，而且还会告诉驱动程序，一旦应用命令，客户端就会收到结果通知。 当 leader 收到状态查询请求时，在 leader 要求所有 peer 确认它仍然是领导者之前，状态驱动程序会收到有关查询的通知（需要满足线性化）。确认结果会被传递给状态机驱动程序，一旦收到多数票，就会对状态机执行查询并将结果返回给客户端。

实际的网络通信由服务器进程处理，这将在单独的部分中进行描述。

RAFT的权衡

单线程状态：所有状态操作都在 leader 的单线程中运行，无法水平可扩展。这里的改进需要运行多个分片 Raft 集群，这超出了项目的范围。

日志复制：只实现了最简单的 Raft 日志复制形式，没有状态快照或快速日志重放。 落后的节点将很难赶上。

集群大小调整：Raft 集群由启动时给定的一组静态节点组成，调整它的大小需要完全重启集群。

查询执行计划中的计划节点 sql::Node 表示关系代数运算符，其中一个节点的输出作为输入流入下一个节点。在上面的示例中，查询首先对 movies 表进行全表扫描，然后将过滤器 released > 2010 应用于行，然后投影（格式化）结果并按评分对其进行排序。

大多数计划都相当简单，将 AST 节点转换为计划节点和表达式。 最棘手的部分是将表名和列名解析为跨越多层别名、连接和投影的结果列索引——这是用 sql::plan::Scope 处理的，它跟踪正在构建的节点可见的名称和将它们映射到列索引。另一个挑战是聚合函数，它被实现为函数参数和分组/隐藏列的预投影，然后是聚合节点，最后是调用最终聚合表达式的后投影——如下例所示：

planner 生成一个非常简单的执行计划，主要关注生成一个正确但不一定快速的执行计划。这意味着它将始终进行全表扫描，始终使用嵌套循环连接等。然后，该计划由一系列实现 sql::Optimizer 的优化器进行优化：

  • ConstantFolder：预评估常量表达式以避免必须为每一行重新评估它们。

  • FilterPushdown：将过滤器更深入地推送到查询中，以减少每个节点评估的行数，例如 通过将单表谓词一直推到表扫描节点，这样过滤的节点就不必穿过 Raft 层。

  • IndexLookup：尽可能将表扫描转换为主键或索引查找。

  • NoopCleaner：尝试删除 noop 操作，例如 过滤评估为常量 TRUE 值的节点。

  • JoinType：将嵌套循环连接转换为等值连接（相等连接谓词）的散列连接。

优化器大量使用布尔代数将表达式转换为更便于使用的形式。 例如，部分过滤器下推（例如跨连接节点）只能下推连接子句（即 AND 部分），因此先将表达式转换为连接范式，以便可以单独考虑每个部分。

执行

每个 SQL plan 节点都有一个对应的执行器，实现了 sql::Executor 特性：

执行器被赋予一个 sql::Transaction 来访问 SQL 存储引擎，并返回一个带有查询结果的 sql::ResultSet。大多数情况下，结果是 sql::ResultSet::Query 类型，其中包含一个列列表和一个行迭代器。 大多数执行器包含用作输入的其他执行器，例如 Filter 执行器通常将 Scan 执行器作为源：

在 sql::Plan 上调用 execute 将构建并执行根节点的执行器，这反过来将递归调用其源执行器（如果有）上的 execute 并处理它们的结果。 执行器通常使用 Rust 的迭代器功能来扩充源返回的行迭代器，例如 通过在其上调用 filter() 并返回一个新的迭代器。 因此，整个执行引擎以流式方式工作，并利用 Rust 的零成本迭代器抽象。

最后，将根 ResultSet 返回给客户端。

服务端

toyDB 服务器使用 Tokio 异步管理 Raft 和 SQL 引擎的网络流量。它在 SQL 客户端的端口 9605 和 Raft 对等端的 9705 端口上打开 TCP 侦听器，两者都使用通过 Serde 编码的 Tokio 流作为协议传递的长度前缀 Bincode 编码消息。

Raft 服务器被拆分为 raft::Server，它运行一个主事件循环在本地 Raft 节点、状态机驱动程序、TCP 对等点和本地状态机客户端（即 Raft SQL 引擎包装器）之间路由 Raft 消息，以及就像定期驱动 Raft 逻辑时钟一样。 它生成单独的 Tokio 任务，维护与所有 Raft 节点的出站 TCP 连接，而内部通信通过 mpsc 通道进行。

SQL 服务器为每个连接的 SQL 客户端生成一个新的 Tokio 任务，从 Raft 顶部的 SQL 存储引擎运行一个单独的 SQL 会话。 它通过 从 sql::Session 发送 server::Request 和 server::Response 消息与客户端通信。

主要的 toydb 二进制文件只是根据命令行参数和配置文件初始化一个 toyDB 服务器，然后通过 Tokio 运行时运行它。

服务端权衡

安全性：所有网络流量都是未经身份验证的明文，安全性被认为超出了该项目的范围。

客户端

toyDB Client 提供了一个简单的 API 用于与服务器交互，主要是通过 execute() 返回 sql::ResultSet 执行 SQL 语句。 它还具有方便的方法 with_txn()，采用闭包执行一系列 SQL 语句，同时自动捕获并重试序列化错误。

还有 client::Pool，它管理一组预连接的客户端，可以检索这些客户端以在多线程应用程序中运行短期查询，而不会产生连接设置成本。

toysql 命令行客户端是一个简单的 REPL 客户端，它使用 toyDB 客户端连接到服务器，并不断提示用户执行 SQL 查询，并显示返回的结果。
